Claude Code Multi-Agent Architecture

Production Readiness Review Swarm

9 specialized agents. 3 gates. Veto authority. Automated production assurance that produces operator-grade artifacts, not reports.

The Problem

Every production change carries risk. Most teams manage that risk through checklists, peer review, and institutional knowledge. But checklists get stale, reviewers get fatigued, and knowledge walks out the door.

The PRR Swarm replaces ad-hoc review with a structured, parallel analysis pipeline. It takes a branch, PR, or diff and produces everything an on-call engineer needs to deploy confidently and recover gracefully.

Architecture: 3-Gate Pipeline

The pipeline enforces separation of duties across three gates. Each gate must complete before the next begins. No agent can self-certify.

Separation of Duties

The architecture enforces a principle borrowed from financial controls: the person who writes the check cannot also approve it. In the PRR Swarm, agents that discover cannot modify, and the validator that judges cannot fix.

Gate 1 — KNOW

Discover What Changed

Three read-only agents run in parallel, each analyzing the diff from a different angle. They cannot modify the codebase — only observe.

Blast Radius Mapper
read-only haiku

Traces every changed file to its L2 dependencies — what calls it, what it calls, what data it touches.

Change Classifier
read-only haiku

Categorizes each change (schema, config, API, feature flag, infra) and assigns a risk tier (T1-T4).

SLA Scanner
read-only haiku

Maps changed code paths to SLA/SLO commitments. Detects implicit SLAs from timeout values and retry configs.

Gate 2 — PLAN

Plan for Failure

Three agents consume Gate 1 findings and produce operational plans. They cannot proceed until all Gate 1 tasks complete.

Rollback Strategist
writes sonnet

Writes executable rollback procedures for every change type. Identifies points of no return.

Observability Auditor
reads sonnet

Audits logging, metrics, and alerting coverage. Flags silent failure paths and dark code.

Failure Analyst
reads sonnet

Runs FMEA — enumerates failure modes, scores each with Risk Priority Numbers (Severity x Occurrence x Detection).

Gate 3 — PROVE

Prove It Works

Two synthesis agents produce operator-facing deliverables. Then the validator reviews everything with veto authority.

Runbook Generator
writes sonnet

Synthesizes all findings into an operator runbook. Passes the 3am rule: executable by on-call who has never seen this code.

Chaos Writer
writes sonnet

Turns FMEA failure modes into testable chaos scenarios with setup, trigger, and pass/fail criteria.

Validator
veto sonnet/opus

QA gate with absolute veto authority. Binary PASS/FAIL. Cannot fix what it finds — only judge.

FMEA Risk Scoring

Each failure mode gets a Risk Priority Number: Severity (1-10) x Occurrence (1-10) x Detection (1-10). Manufacturing reliability methodology, applied to code changes.

0-49
Accept
50-124
Monitor
125-199
Mitigate
200+
BLOCK

Items scoring 200+ automatically trigger a deploy block. The validator verifies that every critical FMEA item has both a rollback procedure and a chaos test scenario.

Not every change needs 9 agents. The swarm triages first, then right-sizes.

Model Tiering

Different cognitive tasks need different model capabilities. Cost matters.

TierModelAgentsWhy
Search haiku Gate 1 mappers Pattern matching and dependency tracing — fast, cheap
Execute sonnet Gate 2+3 agents Synthesis and writing — needs reasoning
Judge opus Validator (full mode) High-stakes judgment calls — worth the cost

Why This Works

Parallel by design

Gate 1 agents run simultaneously. Gate 2 agents run simultaneously. Total wall-clock time is 3 sequential stages, not 9 sequential agents.

Gated, not waterfall

Each gate has explicit blockedBy dependencies. Gate 2 cannot start until Gate 1 completes. The validator cannot start until Gates 2 and 3 complete. No shortcuts.

Veto authority is real

The validator produces a binary PASS/FAIL. It cannot fix what it finds. It cannot negotiate. It holds absolute veto — the separation principle prevents capture.

Artifacts, not reports

The runbook is executable. The chaos scenarios are testable. The rollback plan has verification steps. Everything is designed for the 3am operator, not the morning standup.

Getting Started with Claude Code

Claude Code is Anthropic's CLI tool that gives Claude direct access to your terminal, filesystem, and git. The PRR Swarm runs entirely inside it. Here's how to get started.

Installation

  1. Install Claude Code 
    npm install -g @anthropic-ai/claude-code

    Requires Node.js 18+. Authenticates via your Anthropic API key or through the browser.

  2. Launch in your repo 
    cd your-project && claude

    Claude Code opens an interactive session with full access to your codebase.

  3. Try basic operations first 
    # Ask Claude to explore your codebase
"What are the main entry points in this project?"

# Ask it to make a change
"Add input validation to the /api/users endpoint"

# Ask it to run tests
"Run the test suite and fix any failures"
  4. Run the PRR Swarm 
    # Once agent roles are configured:
/prr-swarm                          # Review current branch vs main
/prr-swarm feature/auth-refactor    # Review specific branch
/prr-swarm #142                     # Review a PR by number

Key Concepts

Skills (slash commands)

Skills are reusable prompts that encode complex workflows. /prr-swarm is a skill that orchestrates the entire 3-gate pipeline. You can create your own skills for any repeatable workflow.

Agent Roles

Each agent has a role file (markdown) that defines its identity, tools, model tier, and anti-patterns. Roles live in ~/.claude/agents/ and are loaded into agents when the swarm spawns them.

Teams and Task Lists

Claude Code can spawn multiple agents that coordinate via shared task lists. Tasks have dependencies (blockedBy), owners, and status tracking. The team lead orchestrates; teammates execute.

CLAUDE.md

Every project can have a CLAUDE.md file with instructions that Claude follows automatically. Think of it as a constitution for your AI assistant — coding standards, routing rules, constraints, and conventions.
