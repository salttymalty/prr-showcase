Claude Code Multi-Agent Architecture

Production Readiness Review Swarm

9 specialized agents. 3 gates. Veto authority. Automated production assurance that produces operator-grade artifacts, not reports.

Getting Started with Claude Code

Claude Code is Anthropic's agentic CLI. You type natural language in your terminal; Claude reads your files, writes code, runs commands, and manages git — with your approval at each step. The PRR Swarm is one example of what you can build with it.

Step 1: Install and Launch

  1. Install Claude Code 
    npm install -g @anthropic-ai/claude-code

    Requires Node.js 18+. On first run, it opens your browser to authenticate with your Anthropic account.

  2. Open a project 
    cd your-project && claude

    Claude Code starts an interactive session. It can see your entire codebase, run terminal commands, and edit files. You approve each action.

  3. Start with plain English 
    "What does this project do? Walk me through the architecture."

    Claude reads your files, traces imports, and explains the structure. This is the fastest way to onboard onto any codebase.

Step 2: Learn the Core Concepts

Claude Code has four building blocks that make the PRR Swarm possible.

CLAUDE.md
your-project/CLAUDE.md

A markdown file at the root of your repo that Claude reads automatically every session. Put your coding standards, project conventions, and any constraints here. Claude will follow them.

Agent Roles
~/.claude/agents/*.md

Markdown files that define specialized agent identities. Each role has a name, model tier, allowed tools, and behavioral constraints. Claude loads these when spawning team members.

Skills
~/.claude/skills/*/SKILL.md

Reusable workflows triggered with /slash-commands. A skill is a markdown file with step-by-step instructions. /prr-swarm is one — you can create your own for any repeatable process.

Teams
~/.claude/teams/

Multi-agent coordination. Claude spawns teammates that share a task list with dependencies. Each teammate runs in its own context with its own role. The lead orchestrates; teammates execute.

Step 3: Try Single-Agent Workflows First

You don't need a 9-agent swarm to get value. Start with one agent doing one thing well.

Explore a codebase you didn't write
"I just cloned this repo. What are the main entry points?
What frameworks does it use? Where are the tests?"

Claude reads files, follows imports, and builds you a mental map. Faster than reading docs.

Write code with context
"Add rate limiting to the /api/upload endpoint.
Follow the same pattern used in /api/auth."

Claude reads the existing pattern, matches the style, writes the implementation, and can run the tests to verify.

Review a PR
"Review PR #47. Focus on error handling and edge cases.
Check if the rollback path is tested."

Claude reads the diff, traces affected code paths, and gives you specific, line-level feedback.

Debug with full filesystem access
"The /api/orders endpoint returns 500 intermittently.
Check the logs, trace the error, and propose a fix."

Claude can read log files, grep for patterns, check config, and trace the issue across files.

Step 4: Set Up Your CLAUDE.md

This is the highest-leverage thing you can do. A good CLAUDE.md means Claude follows your team's conventions automatically instead of you repeating them every session.

Example CLAUDE.md for a production service
# Project: payment-service

## Stack
- Go 1.22, PostgreSQL 15, Redis 7
- Tests: go test with testify assertions
- CI: GitHub Actions, must pass before merge

## Conventions
- All API endpoints must have OpenAPI annotations
- Database queries use sqlc (no raw SQL strings)
- Errors wrap with fmt.Errorf("operation: %w", err)
- All public functions have godoc comments

## Constraints
- No direct DB access from handlers (use service layer)
- No panics in production code (return errors)
- Minimum 80% test coverage on new code
- All migrations must be reversible
Pro tip: Your CLAUDE.md can also include routing rules ("when I mention X, read this file first"), people context ("Ana approves deploys, Josh owns the frontend"), and tool references ("use make lint before committing"). The more context you give, the fewer corrections you'll need.

Quick Setup Script

Or install everything at once:

# Create the directories
mkdir -p ~/.claude/agents
mkdir -p ~/.claude/skills/prr-swarm
mkdir -p ~/.claude/skills/swarm/team-templates

# Download agent roles
BASE="https://salttymalty.github.io/prr-showcase"
for agent in prr-blast-mapper prr-change-classifier prr-sla-scanner \
             prr-rollback-strategist prr-observability-auditor \
             prr-failure-analyst prr-runbook-generator prr-chaos-writer \
             validator; do
  curl -sL "$BASE/agents/$agent.md" -o ~/.claude/agents/$agent.md
done

# Download orchestration files
curl -sL "$BASE/skills/prr-swarm-skill.md" \
     -o ~/.claude/skills/prr-swarm/SKILL.md
curl -sL "$BASE/skills/team-template-prr.md" \
     -o ~/.claude/skills/swarm/team-templates/prr.md

echo "PRR Swarm installed. Run: claude then /prr-swarm"

What Happens When You Run It

Here's the actual flow when you type /prr-swarm in Claude Code:

  1. Claude captures the diff

    It runs git diff main...HEAD to get all changes on your branch, plus a file list and diff stats.

  2. Triage decides the mode

    Less than 5 files? Light mode (3 agents). 5-20 files? Standard (9 agents). Docs only? Skip entirely. No wasted compute.

  3. Gate 1 agents spawn in parallel

    Three read-only agents (Blast Mapper, Classifier, SLA Scanner) analyze the diff simultaneously. They finish in ~2 minutes.

  4. Gate 2 consumes Gate 1 findings

    Rollback Strategist, Observability Auditor, and Failure Analyst read the Gate 1 outputs and produce operational plans.

  5. Gate 3 synthesizes and validates

    Runbook Generator and Chaos Writer produce operator-facing artifacts. Then the Validator reviews everything and issues PASS or FAIL.

  6. You get 8 artifacts and a verdict

    All files land in ~/.claude/prr/{session-id}/. The runbook is ready for your on-call engineer. The chaos scenarios are ready for your test environment.

Adapting this to your stack: The agent role files are plain markdown. You can edit any of them to reference your specific infrastructure — your monitoring tools, your deployment pipeline, your SLA definitions. The agents will use whatever context you give them. Start with the defaults, then customize after your first run.

The Problem

Every production change carries risk. Most teams manage that risk through checklists, peer review, and institutional knowledge. But checklists get stale, reviewers get fatigued, and knowledge walks out the door.

The PRR Swarm replaces ad-hoc review with a structured, parallel analysis pipeline. It takes a branch, PR, or diff and produces everything an on-call engineer needs to deploy confidently and recover gracefully.

Architecture: 3-Gate Pipeline

The pipeline enforces separation of duties across three gates. Each gate must complete before the next begins. No agent can self-certify.

Separation of Duties

The architecture enforces a principle borrowed from financial controls: the person who writes the check cannot also approve it. In the PRR Swarm, agents that discover cannot modify, and the validator that judges cannot fix.

Role Can Read Can Write Can Veto
Gate 1: MappersYesNoNo
Gate 2: PlannersYesYesNo
Gate 3: SynthesizersYesYesNo
ValidatorYesNoYes
Gate 1 — KNOW

Discover What Changed

Three read-only agents run in parallel, each analyzing the diff from a different angle. They cannot modify the codebase — only observe.

Blast Radius Mapper
read-only haiku

Traces every changed file to its L2 dependencies — what calls it, what it calls, what data it touches.

Change Classifier
read-only haiku

Categorizes each change (schema, config, API, feature flag, infra) and assigns a risk tier (T1-T4).

SLA Scanner
read-only haiku

Maps changed code paths to SLA/SLO commitments. Detects implicit SLAs from timeout values and retry configs.

Gate 2 — PLAN

Plan for Failure

Three agents consume Gate 1 findings and produce operational plans. They cannot proceed until all Gate 1 tasks complete.

Rollback Strategist
writes sonnet

Writes executable rollback procedures for every change type. Identifies points of no return.

Observability Auditor
reads sonnet

Audits logging, metrics, and alerting coverage. Flags silent failure paths and dark code.

Failure Analyst
reads sonnet

Runs FMEA — enumerates failure modes, scores each with Risk Priority Numbers (Severity x Occurrence x Detection).

Gate 3 — PROVE

Prove It Works

Two synthesis agents produce operator-facing deliverables. Then the validator reviews everything with veto authority.

Runbook Generator
writes sonnet

Synthesizes all findings into an operator runbook. Passes the 3am rule: executable by on-call who has never seen this code.

Chaos Writer
writes sonnet

Turns FMEA failure modes into testable chaos scenarios with setup, trigger, and pass/fail criteria.

Validator
veto sonnet/opus

QA gate with absolute veto authority. Binary PASS/FAIL. Cannot fix what it finds — only judge.

FMEA Risk Scoring

Each failure mode gets a Risk Priority Number: Severity (1-10) x Occurrence (1-10) x Detection (1-10). Manufacturing reliability methodology, applied to code changes.

0-49
Accept
50-124
Monitor
125-199
Mitigate
200+
BLOCK

Items scoring 200+ automatically trigger a deploy block. The validator verifies that every critical FMEA item has both a rollback procedure and a chaos test scenario.

Model Tiering

Different cognitive tasks need different model capabilities. Cost matters.

TierModelAgentsWhy
Search haiku Gate 1 mappers Pattern matching and dependency tracing — fast, cheap
Execute sonnet Gate 2+3 agents Synthesis and writing — needs reasoning
Judge opus Validator (full mode) High-stakes judgment calls — worth the cost

Why This Works

Parallel by design

Gate 1 agents run simultaneously. Gate 2 agents run simultaneously. Total wall-clock time is 3 sequential stages, not 9 sequential agents.

Gated, not waterfall

Each gate has explicit blockedBy dependencies. Gate 2 cannot start until Gate 1 completes. The validator cannot start until Gates 2 and 3 complete. No shortcuts.

Veto authority is real

The validator produces a binary PASS/FAIL. It cannot fix what it finds. It cannot negotiate. It holds absolute veto — the separation principle prevents capture.

Artifacts, not reports

The runbook is executable. The chaos scenarios are testable. The rollback plan has verification steps. Everything is designed for the 3am operator, not the morning standup.

What the Swarm Caught

The PR author thought this was a clean change — add a column, cap retries at 3, done. The swarm found:

FindingAgentWhy It Matters
BLOCK: Migration locks table on PG < 11 Failure Analyst ADD COLUMN with DEFAULT triggers a full table rewrite on PostgreSQL 10 and below. During rewrite, the payments table is locked. All API calls queue and timeout. The migration script had no version guard.
BLOCK: Redis cache not invalidated Failure Analyst After incrementing retry_count in PostgreSQL, the cached payment object in Redis still shows the old count. API reads cache first, so users see stale data for up to 5 minutes. The Chaos Writer wrote a test that proved it.
WARN: N+1 query in retry worker SLA Scanner + Failure Analyst Worker reads retry_count individually per payment. At 200 payments per batch, that is 200 extra queries every 30 seconds. SLA Scanner flagged the implicit 30s tick interval; Failure Analyst scored the degradation.
WARN: Entangled rollback order Change Classifier Code depends on the column existing. If you rollback the migration before the code, every payment query fails with a missing column error. Classifier flagged this and the Rollback Strategist wrote the correct order.

Real-World Use Cases

Scenarios a production assurance engineer encounters regularly, and how the PRR Swarm handles each one.

Database migration on a high-traffic table

The scenario: A developer adds an index to the orders table (50M rows). They tested on a staging database with 10K rows. It took 200ms. In production, it will lock the table for 4 minutes.

What the swarm finds: The Blast Mapper traces the orders table to every service that reads it. The SLA Scanner flags the p99 latency SLO. The Failure Analyst scores the lock duration against traffic volume. The Rollback Strategist writes a DROP INDEX CONCURRENTLY procedure. The Runbook says: run CREATE INDEX CONCURRENTLY instead, and includes the monitoring queries to watch during the build.

Feature flag rollout with a hidden dependency

The scenario: A feature flag enables a new recommendation engine. The flag is safe to toggle — except the new engine writes to a shared cache key that the checkout service also reads. When the flag turns on, checkout starts showing recommendations instead of cart items.

What the swarm finds: The Blast Mapper traces the cache key to both services. The Change Classifier flags the shared-state dependency. The Failure Analyst identifies the cache collision as a data integrity failure (Severity 8, Detection 9 — nearly invisible until a customer reports it). The Chaos Writer produces a scenario that toggles the flag and verifies checkout still renders correctly.

API deprecation with consumers you forgot about

The scenario: You deprecate GET /api/v1/users/:id in favor of v2. You notified the mobile team and the web team. You forgot about the internal reporting service that calls v1 every hour to build compliance reports.

What the swarm finds: The Blast Mapper traces all callers of the v1 endpoint — including the reporting service found through grep of the endpoint URL across all repos. The SLA Scanner flags the compliance report's implicit SLA (must complete before 6am daily). The Runbook includes a pre-deploy checklist item: verify all consumers have migrated to v2.

Infrastructure change: new load balancer rules

The scenario: DevOps updates the load balancer config to route /api/v2/* to a new service cluster. The regex is slightly wrong — it also matches /api/v2-legacy/*, which should stay on the old cluster.

What the swarm finds: The Change Classifier identifies this as an Infrastructure change (T3 risk). The Blast Mapper traces all URL patterns that match the new regex. The Failure Analyst flags the regex ambiguity and scores it against the SLA for the v2-legacy endpoints. The Chaos Writer produces a scenario that sends requests to every matching URL pattern and verifies they route correctly.

Dependency upgrade with silent behavior change

The scenario: You upgrade your HTTP client library from v2.3 to v2.4. The changelog says "bug fixes." What it doesn't mention: the default timeout changed from 30s to 10s. Your slow upstream API (invoice generation, avg response 12s) starts timing out on every call.

What the swarm finds: The Change Classifier flags dependency upgrades for closer inspection. The Blast Mapper traces all call sites of the HTTP client. The SLA Scanner detects the implicit 30s timeout from the old config and flags the discrepancy with the new default. The Observability Auditor checks whether timeout errors would trigger existing alerts (they would — but only after 5 minutes of accumulated errors).

Midnight config change that nobody reviewed

The scenario: An on-call engineer hotfixes a config value at 2am to stop a bleeding incident. The fix works. They push to main. The next morning, the day team deploys a new feature — the CI pipeline pulls main, which includes the unreviewed config change. The config change conflicts with the new feature.

What the swarm finds: The PRR runs against the full diff from main, not just the feature branch. The Change Classifier catches the config change mixed in with the feature code. The Blast Mapper traces the config value to its consumers. The Runbook flags it: "This deploy includes a config change from commit abc1234 (hotfix by @oncall) that was not part of this feature branch. Verify compatibility before proceeding."
